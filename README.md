## Призначення застосунку
Notes Service — веб-сервіс для керування текстовими нотатками.

Об’єкт нотатки має наступні поля:
- id
- title
- content
- created_at

API сервісу складається з базових ендпоінтів:
- GET /notes — вивести усі нотатки
- POST /notes (title, content) — створити нову нотатку
- GET /notes/<id> — отримати конкретну нотатку за ідентифікатором

## Порт застосунку та конфігурація

Конфігурація передається через аргументи командного рядка (наприклад: --port 8080 --db-host 127.0.0.1 ...).
Порт: 8080 (проксіюється через Nginx на порт 80).

## База даних(V2):
MariaDB Server

## Застосунок та його призначення
Mywebapp (Notes Service) - застосунок для створення, збереження та перегляду текстових нотаток. Допомагає централізовано керувати записами. Застосунок підтримує контент-негоціацію (вивід у форматі JSON або HTML).

## API застосунку

### GET /health/alive та GET /health/ready
Захищені ендпоінти. Згідно з політикою безпеки, Nginx блокує зовнішні запити до технічних шляхів, тому сервер поверне статус 403 (Forbidden).

```bash
curl -i http://127.0.0.1/health/alive
```

### GET /
Поверне список усіх ендпоінтів, які існують в межах сервісу. Формат виводу — HTML.

```bash
curl -i http://127.0.0.1/
```

### GET /notes
Поверне список усіх створених нотаток у форматі JSON (за замовчуванням).

```bash
curl -i http://127.0.0.1/notes
```
Якщо додати заголовок 'Accept: text/html', сервіс віддасть дані у вигляді відмальованої HTML-таблиці.

```bash
curl -i -H 'Accept: text/html' http://127.0.0.1/notes
```
### POST /notes
Дозволяє створити нову нотатку.
```bash
curl -i -X POST http://127.0.0.1/notes -d '{"title":"My First Note", "content":"This is a test note."}' -H 'Content-Type: application/json'
```
### GET /notes/:id
Дозволяє переглянути деталі конкретної нотатки за її ID.
```bash
curl -i http://127.0.0.1/notes/1
```

- Для розробки застосунку та скриптів було використано кодовий редатор VSCode [VSCode](https://code.visualstudio.com/).

- Для підключення до сервера та запуску використовується вбудований термінал або класичний емулятор термінала операційної системи (Bash shell / SSH).

- Тестувати застосунок можна за допомогою утиліти curl в терміналі або у програмі Postman.

## Запуск застосунку

Після успішного встановлення застосунку, systemd автоматично керує запуском сервісу за допомогою [systemd socket activation](https://ilmanzo.github.io/post/systemd-socket-activated-services/) та Nginx як reverse proxy. Застосунок запускається від спеціального системного користувача app без права інтерактивного входу.

Якщо до коду застосунку або конфігурацій було внесено зміни, сервіси потрібно перезапустити:

```bash
sudo systemctl restart mywebapp
sudo systemctl restart nginx
```

## Розгортання застосунку mywebapp
### Встановлення та налаштування операційної системи Linux
1. Для початку потрібно завантажити офіційний образ [Ubuntu 24.04 LTS Server](https://ubuntu.com/download/server), а також програму [VirtualBox](https://www.virtualbox.org/).

2. Створіть нову віртуальну машину (VM) у VirtualBox із такими базовими характеристиками:

VM Name: Ubuntu Server (або Linux)

Base memory: 2048 MB (рекомендовано для комфортної роботи MariaDB та Nginx)

Number of CPUs: 2

Disk Size: 20-30 GB

3. Під час встановлення ОС оберіть стандартні налаштування (LVM-розмітка диска) та обов'язково позначте пункт Install OpenSSH Server для зручного керування терміналом.

4. Створіть дефолтного користувача (наприклад, логін: user, з вашим паролем).

### Автоматичне розгортання (deploy.sh)

5. Увійдіть до системи під дефолтним користувачем. Завантажте код застосунку з репозиторію:

```bash
git clone https://github.com/n1ka13/mywebapp.git
cd mywebapp
```
6. Запустіть скрипт автоматичного розгортання з правами суперкористувача:

```bash
sudo bash deploy.sh
```
7. Що робить скрипт:

- Встановлює системні залежності (Node.js, Nginx, MariaDB).

- Зачищає систему від залишкових конфігурацій попередніх інсталяцій та створює рольових користувачів (student, teacher, operator), присвоюючи їм тимчасові безпечні паролі згідно з політиками PAM операційної системи.

- Налаштовує базу даних mywebapp_db.

- Створює конфіги для Systemd Socket Activation на порту 8080.

- Налаштовує Nginx як reverse proxy на 80 порту з блокуванням шляхів /health/.

- Генерує маркер перевірки варіанта /home/student/gradebook.

- Блокує дефолтного користувача віртуальної машини задля безпеки.

8. Після завершення роботи сценарію первинний обліковий запис встановлення ОС автоматично блокується у системному файлі /etc/shadow. Подальший авторизований доступ здійснюється через створені рольові акаунти (student або teacher) за допомогою встановлених внутрішніх системних паролів проєкту.
Обмежений акаунт operator має право виключно на перезапуск та перевірку статусів веб-сервісу та проксі-сервера Nginx без введення пароля суперкористувача:

```bash
su - operator
sudo systemctl restart mywebapp
```
9. Відкрийте термінал та перевірте статуси сервісів:

```bash
sudo systemctl status mywebapp.socket
sudo systemctl status nginx
```
Вивід має показувати 'active (listening)' та 'active (running)'.

10. Розгортання застосунку завершено. Перевірте коректність створення файлу з оцінкою:

```bash
sudo cat /home/student/gradebook
```
Вивід має показати число 10.

## Коректність роботи налаштованого застосунку
Перевірка здійснюється виконанням команд curl, зазначених у розділі API застосунку. Очікується отримання статусу HTTP 200 OK для бізнес-маршрутів та HTTP 403 Forbidden для технічних маршрутів. Автоматична міграція бази даних виконується під час першого надходження трафіку на сокет.
